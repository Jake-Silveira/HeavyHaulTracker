# FounderOpsAutomation - Workflow Intake Form Project

## Project Overview

This is a professional workflow intake form application designed to collect information about user workflows securely. The application uses a **completely free serverless architecture** with Vercel static hosting and Supabase client-side access secured by Row Level Security (RLS).

### Architecture (v2.0 - Serverless)
```
Browser (User)
   ↓
Static Frontend (Vercel - Free Tier)
   ↓
Supabase Client SDK (JS from CDN)
   ↓
Supabase Services (Free Tier):
  - PostgreSQL Database (500MB)
  - Authentication (50,000 MAU)
  - Row Level Security (RLS)
   ↓
Vercel Serverless Functions (Free - 100k invocations/month):
  - Discord Webhook Notifications (keeps webhook URL secure)
```

**Monthly Cost: $0** 🎉

### Key Features
- ✅ **Zero server costs** - Static files on Vercel, no always-on server
- ✅ **Secure by design** - Row Level Security (RLS) controls database access
- ✅ **Supabase Auth** - Built-in authentication for admin dashboard
- ✅ **Discord notifications** - Via Vercel serverless functions (keeps webhook URL secure)
- ✅ **Auto-redirect** to Carrd site after successful submission
- ✅ **Responsive design** with mobile support
- ✅ **Form validation** and user feedback
- ✅ **Detailed guidance** for users filling out the workflow description
- ✅ **Admin dashboard** with Supabase-based authentication
- ✅ **Cross-origin resource sharing (CORS)** support for Vercel deployments

## Architecture Evolution

### v1.0 (Deprecated) - Server-Proxy Architecture
- ❌ Required always-on Express server on Railway/Render
- ❌ Used Supabase service role key (secret, never exposed)
- ❌ Custom admin_users table with bcrypt password hashing
- ❌ Server-side session management
- ❌ Ongoing hosting costs

### v2.0 (Current) - Serverless Architecture
- ✅ Static files on Vercel (no server needed)
- ✅ Uses Supabase anon key (safe to expose, restricted by RLS)
- ✅ Supabase Auth for admin login (no custom auth needed)
- ✅ Row Level Security policies control all database access
- ✅ Completely free within tier limits!

## Building and Running

### Prerequisites
- Vercel account (free) - [Sign up](https://vercel.com)

### Setup

1. **Configure Supabase:**
   - Create a project
   - Run `sql-migration.sql` in Supabase SQL Editor
   - Create admin user via Authentication → Users
   - Add admin user to `admin_users` table

2. **Deploy to Vercel:**
   - Push code to GitHub
   - Connect to Vercel
   - Set environment variables (see below)
   - Deploy!

3. **Local Development:**
   ```bash
   npm install
   cp .env.example .env
   # Edit .env with your Supabase credentials
   node build.js  # Inject env vars into HTML
   npx serve .    # Serve locally
   ```

### Required Supabase Table Schema

Run this in Supabase SQL Editor:

```sql
-- Main submissions table
CREATE TABLE intake_submissions (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT NOT NULL,
  workflow TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Admin users table
-- Drop old table first if migrating from v1
DROP TABLE IF EXISTS admin_users CASCADE;

CREATE TABLE admin_users (
  id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  role TEXT NOT NULL DEFAULT 'admin',
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Enable Row Level Security
ALTER TABLE intake_submissions ENABLE ROW LEVEL SECURITY;
ALTER TABLE admin_users ENABLE ROW LEVEL SECURITY;

-- Grant table permissions for client-side SDK usage
GRANT INSERT, SELECT ON intake_submissions TO anon;
GRANT SELECT ON admin_users TO authenticated;

-- RLS Policies for intake_submissions
CREATE POLICY "public_insert_submissions" ON intake_submissions
FOR INSERT TO anon WITH CHECK (true);

CREATE POLICY "public_select_submissions" ON intake_submissions
FOR SELECT TO anon USING (true);

CREATE POLICY "admin_select_submissions" ON intake_submissions
FOR SELECT TO authenticated USING (true);

CREATE POLICY "service_role_full_access_intake_submissions" ON intake_submissions
FOR ALL TO service_role USING (true) WITH CHECK (true);

-- RLS Policies for admin_users
CREATE POLICY "admin_users_select_any_admin" ON admin_users
FOR SELECT TO authenticated USING (true);
```

### Required Environment Variables

Set these in Vercel Dashboard → Settings → Environment Variables:

| Variable | Description | Example |
|----------|-------------|---------|
| `SUPABASE_URL` | Supabase project URL | `https://xxx.supabase.co` |
| `SUPABASE_ANON_KEY` | Supabase anon public key | `eyJhbGc...` |
| `DISCORD_WEBHOOK_URL` | Discord webhook URL (optional) | `https://discord.com/api/webhooks/...` |

**Important:** 
- Use `SUPABASE_ANON_KEY` (not service role key!)
- The anon key is safe to expose in browser because RLS restricts access
- Never use service role key in this architecture

## Admin Dashboard Setup

The admin dashboard is accessible at `/admin` and uses Supabase Authentication:

### Setting Up Admin Access

1. **Create Admin User in Supabase:**
   - Go to Authentication → Users in Supabase Dashboard
   - Click "Add User" → "Create new user"
   - Enter email and set a strong password
   - Copy the User ID (UUID)

2. **Add to Admin Users:**
   ```sql
   INSERT INTO admin_users (id, email, role) 
   VALUES ('UUID_FROM_SUPABASE_AUTH', 'admin@example.com', 'admin');
   ```

3. **Login:**
   - Visit `/admin` on your deployed site
   - Login with the email/password you created

### Admin Features
- Secure login via Supabase Auth
- View all workflow submissions in a table format
- See submission details (ID, name, email, workflow, date)
- Click on truncated workflow descriptions to view the full text
- Track submission statistics (total submissions, today's submissions)
- Automatic session management via Supabase Auth

## Security Measures

This application uses multiple security layers:

### 1. Row Level Security (RLS)
- **Database-level access control** - Even with the anon key, users can only perform allowed operations
- Anonymous users can only INSERT (submit forms)
- Authenticated users can SELECT (view submissions)
- Service role has full access (for serverless functions)

### 2. Supabase Anon Key
- **Safe to expose in browser** - Unlike service role key
- Restricted by RLS policies
- Cannot bypass database security
- Only allows operations permitted by RLS

### 3. Supabase Auth
- Built-in password hashing
- Session management handled securely
- Token-based authentication
- Automatic token refresh

### 4. Input Sanitization
- Client-side input validation
- Script tag detection and removal
- HTML tag stripping
- Email format validation

### 5. Serverless Functions
- Discord webhook URL kept secure in Vercel env vars
- Never exposed to browser
- Only accessible via `/api/discord-webhook` endpoint

## Troubleshooting

### Form Submission Fails
- Check Supabase URL and anon key are correct in Vercel env vars
- Verify RLS policies are enabled and configured
- Check browser console for error messages
- Review Supabase Table Editor to see if data is being inserted

### Admin Login Fails
- Verify user exists in Supabase Auth (Authentication → Users)
- Check user is added to `admin_users` table with correct UUID
- Ensure password is correct

### Discord Notifications Not Working
- Verify `DISCORD_WEBHOOK_URL` is set in Vercel env vars
- Check webhook URL is valid and active in Discord
- Review Vercel function logs (Dashboard → Functions → Logs)

See `VERCEL_SETUP.md` for comprehensive troubleshooting guide.
