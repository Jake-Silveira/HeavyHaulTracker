# Vercel + Supabase Deployment Guide

## Architecture Overview

This project uses a **completely free serverless architecture**:

```
Browser (User)
   ↓
Static Frontend (Vercel - Free Tier)
   ↓
Supabase Client (JS SDK from CDN)
   ↓
Supabase Services (Free Tier):
  - PostgreSQL Database (500MB)
  - Auth (50,000 MAU)
  - Row Level Security (RLS)
   ↓
Vercel Serverless Functions (Free Tier - 100k invocations/month):
  - Discord Webhook Notifications
```

**Cost: $0/month** (within free tier limits)

---

## What Changed from the Old Architecture

### Before (Server-Proxy Architecture)
- ❌ Node.js/Express server running on Railway/Render
- ❌ Service Role Key (secret, never exposed)
- ❌ Server-side sessions for admin auth
- ❌ Custom `admin_users` table with bcrypt
- ❌ Server handles all database operations
- ✅ Costs money to host server

### After (Client-Side Supabase)
- ✅ Static files on Vercel (no server needed)
- ✅ Anon Key (safe to expose, restricted by RLS)
- ✅ Supabase Auth for admin login
- ✅ Row Level Security policies control access
- ✅ Browser talks directly to Supabase
- ✅ Completely free!

## Step-by-Step Deployment

### Step 1: Set Up Supabase

#### 1.1 Create a Supabase Project
1. Go to [Supabase Dashboard](https://app.supabase.com)
2. Create a new project
3. Note your **Project URL** and **Anon Key** (Settings → API)

#### 1.2 Create Database Tables

Run this SQL in the Supabase SQL Editor:

```sql
-- Main submissions table
CREATE TABLE intake_submissions (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT NOT NULL,
  workflow TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Admin users table (tracks which users can access dashboard)
-- Drop old table first if migrating from v1
DROP TABLE IF EXISTS admin_users CASCADE;

CREATE TABLE admin_users (
  id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  role TEXT NOT NULL DEFAULT 'admin',
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

#### 1.3 Enable Row Level Security (RLS)

Run the migration script: `sql-migration.sql`

Or run this directly:

```sql
-- Enable RLS
ALTER TABLE intake_submissions ENABLE ROW LEVEL SECURITY;

-- IMPORTANT: Grant table permissions to anon role for client-side SDK
GRANT INSERT, SELECT ON intake_submissions TO anon;

-- Public can submit (INSERT)
CREATE POLICY "public_insert_submissions" 
ON intake_submissions 
FOR INSERT 
TO anon 
WITH CHECK (true);

-- Public can SELECT (needed for .insert().select() pattern in client SDK)
CREATE POLICY "public_select_submissions" 
ON intake_submissions 
FOR SELECT 
TO anon 
USING (true);

-- Authenticated admins can view submissions
CREATE POLICY "admin_select_submissions" 
ON intake_submissions 
FOR SELECT 
TO authenticated 
USING (true);

-- Service role has full access (for serverless functions)
CREATE POLICY "service_role_full_access_intake_submissions" 
ON intake_submissions 
FOR ALL 
TO service_role 
USING (true) 
WITH CHECK (true);

-- Enable RLS on admin_users
ALTER TABLE admin_users ENABLE ROW LEVEL SECURITY;

-- Authenticated users can view admin users
CREATE POLICY "admin_users_select_any_admin" 
ON admin_users 
FOR SELECT 
TO authenticated 
USING (true);
```

#### 1.4 Create Admin User via Supabase Auth

1. Go to **Authentication → Users** in Supabase Dashboard
2. Click **Add User** → **Create new user**
3. Enter admin email and set a **strong password**
4. Click **Create user**
5. Copy the **User ID** (UUID)

Now add this user to the `admin_users` table:

```sql
-- Replace with the actual UUID from Supabase Auth
INSERT INTO admin_users (id, email, role) 
VALUES ('PASTE_UUID_HERE', 'admin@example.com', 'admin');
```

#### 1.5 Verify Setup

Run this to check your RLS policies:

```sql
SELECT 
  tablename,
  policyname,
  cmd,
  roles,
  qual
FROM pg_policies 
WHERE tablename IN ('intake_submissions', 'admin_users');
```

### Step 3: Deploy to Vercel

#### 3.1 Connect Your Repository

**Option A: Deploy via Vercel CLI**
```bash
# Install Vercel CLI
npm install -g vercel

# Login to Vercel
vercel login

# Deploy (from project root)
vercel

# Follow prompts to set up project
# - Set up and deploy? Y
# - Which scope? (select your account)
# - Link to existing project? N
# - Project name? (enter name)
# - Directory? ./
# - Override settings? N
```

**Option B: Deploy via GitHub Integration**
1. Go to [Vercel Dashboard](https://vercel.com/dashboard)
2. Click **Add New...** → **Project**
3. Import your GitHub repository
4. Configure environment variables (as above)
5. Click **Deploy**

#### 3.2 Deploy to Production

Once you've tested and verified everything works:

```bash
vercel --prod
```

Or via GitHub: Push to your `main` branch (if configured).

## Security Model

### How We Keep Things Secure Without a Server

1. **Row Level Security (RLS)** - Database-level access control
   - Anonymous users can only INSERT (submit forms)
   - Authenticated users can SELECT (view submissions)
   - Service role has full access (for serverless functions)

2. **Supabase Anon Key** - Safe to expose in browser
   - Restricted by RLS policies
   - Cannot bypass database security
   - Only allows operations permitted by RLS

3. **Supabase Auth** - Secure admin authentication
   - Password hashing handled by Supabase
   - Session management built-in
   - Token-based authentication

4. **Discord Webhook URL** - Kept secure in Vercel env vars
   - Only accessible via serverless function
   - Never exposed to browser
   - Rate-limited by Discord

## Troubleshooting

### Form Submission Fails

**Check:**
1. Supabase URL and Anon Key are correct in Vercel env vars
2. RLS policies are enabled and configured correctly
3. Browser console for error messages
4. Supabase Table Editor to see if data is being inserted

**Fix:**
```sql
-- Verify RLS is enabled
SELECT tablename, rowsecurity FROM pg_tables WHERE tablename = 'intake_submissions';

-- Verify policies exist
SELECT * FROM pg_policies WHERE tablename = 'intake_submissions';
```

### Admin Login Fails

**Check:**
1. User exists in Supabase Auth (Authentication → Users)
2. User is added to `admin_users` table with correct UUID
3. Password is correct

**Fix:**
```sql
-- Check if admin user exists in auth.users
SELECT id, email FROM auth.users WHERE email = 'admin@example.com';

-- Check if admin profile exists
SELECT * FROM admin_users WHERE email = 'admin@example.com';

-- If missing, re-add the admin profile with correct UUID
INSERT INTO admin_users (id, email, role) 
VALUES ('UUID_FROM_AUTH_USERS', 'admin@example.com', 'admin');
```

### Discord Notifications Not Working

**Check:**
1. `DISCORD_WEBHOOK_URL` is set in Vercel env vars
2. Webhook URL is valid and active in Discord
3. Vercel function logs (Vercel Dashboard → Functions → Logs)

**Fix:**
- Test webhook directly: `curl -X POST -H "Content-Type: application/json" -d '{"content":"test"}' YOUR_WEBHOOK_URL`
- Check Vercel function logs for errors

## Migration from Old Architecture

If you're migrating from the server-proxy architecture:

1. **Run the SQL migration**: `sql-migration.sql`
2. **Create Supabase Auth users** for your admins
3. **Update environment variables** (use ANON_KEY instead of SERVICE_ROLE_KEY)
4. **Deploy to Vercel** (no more Railway/Render server needed!)
5. **Test everything** before decommissioning old server
